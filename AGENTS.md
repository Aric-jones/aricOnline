# AGENTS.md

本仓库采用文档驱动开发。

## 优先级

1. `AGENTS.md`
2. `docs/README.md`
3. `docs/project-overview.md`
4. `docs/modules.md`
5. `docs/requirements/*.md`
6. `docs/changelog.md`
7. `README.md` 和各模块 README
8. 代码注释和实现细节

## 核心规则

- 没有对应需求文档，不开始改代码。
- 任何功能、新增、修复、重构、权限变更、路由变更、接口变更、数据库变更、部署变更，都必须有需求文档。
- 每次代码变更都必须能追溯到某份需求文档或一条已记录的问题修复。
- 代码改完后，必须更新 `docs/changelog.md`。
- 如果改动影响模块边界、职责或行为，必须更新 `docs/modules.md`。
- 如果改动影响产品定位、用户角色或系统范围，必须更新 `docs/project-overview.md`。

## 必须执行的流程

1. 先看需求，定位受影响范围。
2. 在 `docs/requirements/` 下新建或更新对应需求文档。
3. 再实现代码改动。
4. 再更新 `docs/changelog.md`。
5. 如果改动改变了模块含义或系统范围，同步更新 `docs/modules.md` 或 `docs/project-overview.md`。
6. 用最小必要的测试或构建步骤验证结果。

## 需求文档规则

- 文件名使用日期格式：`YYYY-MM-DD-short-name.md`。
- 每份需求文档必须包含：
  - 背景
  - 目标
  - 非目标
  - 影响范围
  - 受影响文件
  - 验收标准
  - 测试计划
  - 风险与回滚
- 如果涉及路由、菜单、权限、接口、数据、配置或部署，必须写影响范围表。

## 变更日志规则

- `docs/changelog.md` 是已完成变更的唯一记录入口。
- 每条变更都要记录：
  - 日期
  - 标题
  - 类型
  - 关联需求文档
  - 影响范围
  - 验证说明
  - 回滚说明

## 文档边界

- `README.md` 只保留高层介绍和入口。
- 详细的产品功能和模块职责放在 `docs/`。
- 部署说明保留在 `deploy/README.md`。
- 前端和后台各自的专用说明，只在确实需要时写到对应模块的 README 中。

## 编辑纪律

- 变更范围要紧贴请求本身。
- 不要重写无关文件。
- 保留当前未完成的工作，除非请求明确要求替换。
- 如果请求有歧义，先通过仓库内容确认，再把采用的假设写清楚。

## 交付要求

- 每次任务完成后，除了正常总结外，都要额外给出一条中文的 Git 提交文案。
- 提交文案采用 Conventional Commits 风格：`type(scope): 中文摘要`。
- 类型优先使用：`init`、`feat`、`fix`、`docs`、`refactor`、`perf`、`test`、`chore`、`build`、`ci`、`revert`。
- 摘要要简洁、明确、可直接用于提交，不写空话。
- 如果本次变更包含多个方面，优先给出一条主提交文案，必要时再补一条备选文案。
